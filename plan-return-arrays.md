# Plan: return-array allocation in `seamless-signature`

## Context

Follow-up to the main [seamless-signature plan](idempotent-sprouting-kernighan.md). That plan describes the YAML schema language and C header generator but glosses over how output arrays are handled. This file captures the missing piece.

Seamless owns all memory for output arrays. A compiled function matching a signature generated from this schema **must not** allocate the storage that backs any output array — anything `malloc`'d inside the function is purely scratch and must be `free`'d before the function returns. The caller (Seamless) pre-allocates every output buffer and passes it in as a pointer, and the function is expected to fill it. This dogma is load-bearing for reproducible caching and for crossing the C ↔ Python boundary without ownership ambiguity.

This splits output arrays into two cases.

## Case A — static-sized output (easy)

Every dim in the output `shape` is either a literal int or a wildcard that already appears in the `inputs` section. The caller knows the full output length before the call, pre-allocates it, and passes the pointer in. No schema extension needed beyond what the main plan already specifies.

Example test fixture — `tests/fixtures/static_return.yaml`:

```yaml
function_name: pairwise_distance
inputs:
  - {name: coords, dtype: float32, shape: [N, 3]}
outputs:
  - {name: dist,   dtype: float32, shape: [N, N]}   # N comes from inputs
```

## Case B — dynamic-sized output (the hard one)

The output `shape` contains a wildcard that does **not** appear in any input — its value only becomes known *as the function runs*. The function still must not allocate the output: somebody has to tell Seamless how large the buffer has to be before the call.

Example test fixture — `tests/fixtures/dynamic_return.yaml`:

```yaml
function_name: find_peaks
inputs:
  - name: signal
    dtype: float32
    shape: [N]
  - name: threshold
    dtype: float32
outputs:
  - name: peak_positions
    dtype: int32
    shape: [K]          # K is a new wildcard, not present in inputs
  - name: peak_values
    dtype: float32
    shape: [K]          # same K — shared by name, just like input wildcards
```

Here `N` is known from `signal.shape[0]` at call time, but `K` (the number of peaks found) is only knowable after the computation. Seamless cannot pre-allocate `peak_positions` / `peak_values` without further information.

## Mechanism

No YAML schema extension is needed — the existing syntax already expresses everything. The mechanism lives in three places: classification of wildcard dims, the C header emission rule, and the Python wrapper generated by the Seamless runtime glue.

### Wildcard classification

After a `Signature` is parsed, its wildcard dim names partition into two disjoint sets:

- **input-derived**: appears in the shape of at least one `inputs[*]` parameter.
- **output-only**: appears only in `outputs[*]` parameters, never in any input.

The `Signature` dataclass grows two derived properties:

```python
@property
def input_wildcards(self) -> tuple[str, ...]: ...
@property
def output_wildcards(self) -> tuple[str, ...]: ...
```

Both preserve first-seen order. A name cannot be in both sets — a collision is a `TypeError` at `__post_init__` time.

### C header emission

For each output-only wildcard `K`, the generated C signature carries **two** extra parameters:

1. `unsigned int maxK` — caller-provided upper bound on `K`, declared alongside the other input integers.
2. `unsigned int *K` — written by the function with the actual count. Declared alongside the other output pointers.

The function uses `maxK` to bail out with a nonzero return code if it would otherwise overflow, and writes the final count through `*K` (which must satisfy `*K <= maxK`).

**Full `transform()` argument order** (supersedes step 3 of C header generation in the main plan):

1. Input-derived wildcard ints — one `unsigned int <name>` per unique name.
2. Output-only wildcard capacity bounds — one `unsigned int max<Name>` per unique name.
3. Each input parameter in YAML order — scalar by value, or `const <elem_t> *` for arrays.
4. Output-only wildcard actual-count pointers — one `unsigned int *<name>` per unique name.
5. Each output parameter in YAML order — scalar `<ctype> *` or `<elem_t> *` for arrays.

The `max<Name>` / `*<Name>` names are produced by literal string concatenation (`"max" + name` / `name`) so the user's YAML naming is preserved verbatim.

### Example: generated header for `find_peaks`

From the `dynamic_return.yaml` fixture above:

```c
/* Auto-generated from dynamic_return.yaml — do not edit. */
#include <stdint.h>
#include <stdbool.h>

int transform(
    unsigned int N,
    unsigned int maxK,
    const float *signal,
    float threshold,
    unsigned int *K,
    int32_t *peak_positions,
    float *peak_values
);
```

Contract for the function author:

- Reads `N`, `maxK`, `signal`, `threshold`.
- Writes at most `maxK` entries into `peak_positions` and `peak_values`.
- Writes the actual count through `*K`, and must satisfy `*K <= maxK`.
- Returns `0` on success, nonzero on error (including "more than `maxK` results would be produced").
- Any scratch memory `malloc`'d inside the function must be `free`'d before return. The function does **not** allocate `peak_positions` or `peak_values`.

### Python wrapper

The Seamless runtime glue (not this package — `seamless-signature` only generates the C header and exposes the classification properties on `Signature`) produces a Python-facing wrapper whose signature is:

- Inputs positionally in YAML order, with input-derived wildcard ints **elided** — they come from `input.shape[i]` automatically.
- One required keyword-only argument `max<Name>` per output-only wildcard.

For `find_peaks`:

```python
def find_peaks(signal, threshold, *, maxK): ...
```

Wrapper behavior on call:

1. Derive input-derived wildcards from argument `.shape` (here, `N = signal.shape[0]`).
2. Pre-allocate each output array with output-only wildcards substituted by their `max<Name>` values — here, `np.empty((maxK,), dtype=np.int32)` and `np.empty((maxK,), dtype=np.float32)`.
3. Allocate one `uint` scratch cell per output-only wildcard for the written-back count.
4. Call the C `transform(...)` with all pointers wired up.
5. On success, read back each written count (here, `K`) and trim each output array down to the actual shape according to the global **trim mode** setting (see next subsection). The caller receives an array of shape `(K, ...)`; `K` is implicit in that shape and not returned separately.
6. On nonzero return code, raise a Seamless-level error reporting the `max<Name>` values so the caller can retry with larger bounds.

### Trim mode (global setting)

Shrinking the oversized `maxK` buffer down to the actual `K` is controlled by a global Seamless setting `dynamic_output_trim_mode` with two values:

- **`"copy"`** — default, safe. `np.array(arr[:K])`. Allocates a fresh buffer of the true size, copies the first `K` elements, and lets the oversized original go out of scope. Peak memory briefly holds both buffers during the copy; behavior is defined regardless of any stray Python- or C-level references to the original.
- **`"inplace_unsafe"`** — opt-in, fast. `arr.resize((K, ...), refcheck=False)` after explicitly dropping any binding intermediaries from the C call. Shrinks via `PyDataMem_RENEW` (realloc) with no copy and no doubled peak. **Unsafe in the C sense**: if any Python- or C-level reference to the buffer outlives the resize, the realloc may move the buffer and those references become use-after-free. Only flip this on after profiling shows the copy is a real cost and the wrapper's reference discipline has been audited.

Default is `"copy"`. The setting lives as a module-level variable in the Seamless runtime glue — exact storage (plain module attribute, `seamless-config` YAML entry, environment variable, or all three) is the runtime glue's concern and outside `seamless-signature`'s scope. `seamless-signature` itself does not read or act on this setting; it is documented here only because the wrapper-generation contract the C header implies has to be described somewhere.

### Mixed output shapes

The trailing-fixed rule from the main plan still applies. Examples:

- `shape: [K, 3]` with `K` output-only → element typedef `float32_3[3]`, C parameter `float32_3 *name`, pre-allocation shape `(maxK, 3)`, trim to `result[:K]`.
- `shape: [K, N]` with `K` output-only and `N` input-derived → pre-allocation `(maxK, N)`, trim to `result[:K]`. `N` is fine as a leading wildcard in element-shape terms because it's known at call time.
- `shape: [N, K]` (input-derived outermost, output-only innermost) — pre-allocation `(N, maxK)`, trim to `result[:, :K]`. Non-contiguous sub-view, still valid.

## Scope of changes to the main plan

The main [seamless-signature plan](idempotent-sprouting-kernighan.md) needs these amendments once this sub-plan is accepted:

- **Dataclasses**: add `input_wildcards` / `output_wildcards` properties on `Signature`, and enforce the input-vs-output-only disjointness in `__post_init__`.
- **C header generation algorithm**: extend the argument ordering to the 5-step sequence above (currently the main plan has a 3-bullet rule that only handles input-derived wildcards).
- **Tests**: add `static_return.yaml` and `dynamic_return.yaml` fixtures plus golden-header comparisons; add negative case for a name that is both input-derived and output-only.
- **Explicit non-goals**: nothing to remove; dynamic-return support is in scope for iteration 1 under this sub-plan.
